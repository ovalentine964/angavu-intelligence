## Architecture

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Msaidizi    │────▶│   Backend    │────▶│   OpenWA     │────▶│  WhatsApp    │
│  App         │     │   (Python)   │     │  (Baileys)   │     │  (End User)  │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
       │                    │                    │
       │  Onboarding        │  Webhook           │  Whisper STT
       │  WhatsApp Setup    │  /api/v1/webhooks  │  Voice Transcription
       │                    │                    │
       └────────────────────┴────────────────────┘
                   Report Scheduler
                   (Backend Cron)
```

**Key Design Decision:** WhatsApp is a **pure report delivery channel**. Transaction recording happens in the Msaidizi app (offline). Reports are generated by the cloud and delivered via WhatsApp when workers are online.

## Integration Flow

### Onboarding (App → Backend → WhatsApp)

1. **BusinessDiscoveryFragment** conversation reaches `AskChallenge`
2. Next step: `AskWhatsApp` — "Unatumia WhatsApp?"
3. If yes: `ConfirmWhatsAppNumber` — "Namba yako ya WhatsApp ni ipi?"
4. Phone extracted, normalized to 254XXXXXXXXX format
5. `WhatsAppConnected` — "Msaidizi imeunganishwa! Ripoti ya kwanza itafika saa 7 usiku."
6. Backend registers WhatsApp connection via `/api/v1/whatsapp/connect`
7. Test message sent via OpenWA

### Report Delivery (Backend → OpenWA → WhatsApp)

1. **Backend scheduler** triggers at configured time (7 PM EAT daily, etc.)
2. Calls `/api/v1/webhooks/whatsapp/daily-reports`
3. `WhatsAppDelivery.send_daily_reports_to_all()` iterates active WhatsApp users
4. For each user: generates report text + chart image
5. Sends via OpenWA `/send-message` (text) or `/send-image` (chart + caption)
6. For voice reports: generates TTS audio, sends via `/send-voice`

### Incoming Messages (WhatsApp → OpenWA → Backend)

1. User sends "Ripoti ya leo" via WhatsApp
2. OpenWA receives via Baileys `messages.upsert` event
3. OpenWA transcribes voice via Whisper if needed
4. Forwards to backend `/api/v1/webhooks/whatsapp` with HMAC signature
5. Backend validates signature, routes to `WhatsAppBot.process_message()`
6. Bot parses command, generates report, sends back via OpenWA

## Health Monitoring

### OpenWA Health Endpoints

- `GET /health` — Full health status (service, WhatsApp, stats)
- `GET /status` — Quick connection status
- `GET /qr` — QR code for authentication

### Backend Health Proxy

- `GET /api/v1/webhooks/whatsapp/openwa-health` — Proxies to OpenWA health

### Monitoring Service (`whatsapp_health.py`)

- `check_health()` — Checks OpenWA HTTP + WhatsApp session
- `should_alert()` — Returns true after 2+ consecutive failures or logout
- `alert_admin()` — Logs critical alert (integrate with SMS/email in production)

### Container Health Checks (Docker)

- OpenWA: `wget http://localhost:3000/health` every 30s
- Whisper: `wget http://localhost:9000/health` every 30s (60s start period)

## Key Bug Fixes

1. **Phone encryption bug** — `send_report` endpoint was passing encrypted phone to `send_message`. Now decrypts first.
2. **Missing endpoints** — Backend called `/send-image` but OpenWA only had `/send-media`. Added dedicated `/send-image` and `/send-voice` endpoints.
3. **Wrong API URL** — Trigger was calling `/api/send-message` instead of `/send-message`.
4. **Fragile cron** — OpenWA had `setInterval` for daily reports (timezone drift, duplicate sends). Now backend-driven.
5. **No retry logic** — Failed messages now retry 3 times with exponential backoff.
6. **No rate limiting** — Added in-memory rate limiter (100 req/min per IP).
7. **Help text wrong time** — Daily report time shown as "saa 1 jioni" (1 PM), corrected to "saa 7 usiku" (7 PM).

## Security

- **HMAC signatures** on all OpenWA → Backend webhook calls
- **Phone numbers** stored encrypted (`phone_encrypted`) with hash lookup (`phone_hash`)
- **Rate limiting** on OpenWA API (100 req/min per IP)
- **Webhook secret** required (env var, min 16 chars)
- **Auth state** persisted in Docker volume (survives container restarts)
- **No phone exposure** in API responses (masked as `****`)
